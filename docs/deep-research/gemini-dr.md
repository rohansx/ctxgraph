Comprehensive Architectural Framework and Optimization Strategy for Edge-Native Agentic Context EnginesOperational Baseline and Crate Layout of the ctxgraph EngineThe execution of long-term episodic memory for autonomous agents requires a strict departure from cloud-dependent context parsing pipelines. The ctxgraph engine functions as an open-source, privacy-first context orchestration system designed to extract highly structured, schema-typed entities and relations directly from unstructured text streams. It exposes an environment distributed via native package management, a standardized Model Context Protocol (MCP) hosting node, and a modular Rust software development kit (crates/ctxgraph-core). The codebase is organized as a unified workspace comprising approximately 12,000 lines of Rust code. The underlying architecture spans two primary layers: a bi-temporal storage infrastructure and a multi-tiered information extraction execution framework.The storage layer models data inside an embedded database backend using an optimized schema. The database file manages five core tables: episodes for raw text blocks, entities for tracking object definitions, aliases for linking alternative references via Jaro-Winkler string similarity matrices, embeddings storing 384-dimensional dense vectors, and edges modeling facts with strict bi-temporal validity coordinates. Retrieval across this graph does not rely on iterative graph traversals; instead, the engine uses a Reciprocal Rank Fusion query routing architecture. This structure combines Full-Text Search (FTS5) indices, vector space cosine similarity matrices, and recursive common table expressions (CTEs) within a single execution cycle, yielding comprehensive context resolution in less than 15 milliseconds.The workspace layout isolates operational domains into discrete sub-crates to facilitate concurrent development and testing:crates/

The data flow through the information processing pipeline follows a tiered execution sequence designed to prioritize local execution before escalating to expensive remote resources. Tier 1 executes a local, zero-cost transformer pipeline on every incoming text block. It processes text through a local token classification engine for object tracking, runs cross-sentence coreference resolution to replace pronouns, applies dictionary additions, executes type remapping, and strips grammatical artifacts.Following this local pass, data encounters a strict confidence gate. The pipeline bypasses subsequent processing if the local output complies with the schema. However, the gate triggers escalation to Tier 2 if it identifies any of the following threshold conditions: an entity density falling below 1.5 objects per 10 words, an average token assignment confidence score below 0.4, more than 40% of extracted objects failing to map onto the active schema taxonomy, extreme data sparseness (defined as an episode exceeding 25 words with fewer than 5 unique entities), or the presence of complex system identifiers such as @, v2, ::, or outage.Tier 2 routes data to a local inference server instance via automated port identification. The engine cycles through a priority queue of local configurations to optimize execution quality:$$\text{gemma3n:e4b} \rightarrow \text{gemma4:e4b} \rightarrow \text{gemma3n:e2b} \rightarrow \text{gemma4:e2b} \rightarrow \text{llama3.2:3b}$$If local infrastructure is unavailable, processing transitions to Tier 3, executing a single-call schema-constrained JSON extraction pass against an OpenAI-compatible cloud endpoint defaulting to gpt-4o-mini. To preserve privacy during remote operations, an anonymization layer strips personally identifiable information (PII) from the text stream prior to cloud transmission when specific feature flags are enabled. When an extraction engine processes sequential data streams, it must repeatedly evaluate a static system prompt containing complex schema definitions, task constraints, and few-shot formatting examples. In a baseline stateless setup, the engine must recompute Query, Key, and Value (KV) tensors for this entire prefix on every request, creating an execution bottleneck that increases Time to First Token (TTFT) and consumes excessive GPU cycles.To eliminate this computational overhead, modern inference servers like llama.cpp and Ollama implement prefix-based prompt caching. This mechanism retains the pre-computed KV tensors of static prompt blocks in memory, allowing subsequent inference requests to skip the prefill computation and resume generation directly from the cached state.In llama-server, host-memory prompt caching offloads pre-computed KV tensors from limited GPU VRAM into system RAM (host memory).
